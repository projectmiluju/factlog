# 개발 일지: Web UI MVP 구현과 테스트 환경 복구

**일자:** 2026-04-04
**관련 버전:** v0.1.0
**관련 역할:** build, qa

## 배경 (Context)
`#4` 이슈의 목표는 FactLog의 핵심 사용자 플로우를 브라우저에서 끝까지 시연 가능한 형태로 만드는 것이었다. 백엔드 API만으로는 채용 과제의 설득력이 약하기 때문에, 입력 화면, 분석 결과, 조치 기록, 대시보드 요약을 하나의 데모 흐름으로 연결할 필요가 있었다.

## 문제 (Problem)
저장소에는 React 프론트엔드가 아예 없었고, 기존 FastAPI API도 UI가 기대하는 `조치 기록 저장`, `최근 이력 조회`, `검증 요약 조회`를 모두 제공하지 않았다. 추가로 UI 테스트를 붙이자 Recharts의 `ResponsiveContainer`가 JSDOM에서 `ResizeObserver`를 요구해 테스트가 전부 실패했다.

## 시도한 것들 (Attempts)
1. **시도 1:** `web/`에 Vite + React + TypeScript + Recharts를 새로 스캐폴드하고, 최소 스타일만으로 화면을 연결했다.
   시연용 UI를 빠르게 만들 수 있었지만, 백엔드가 조치 저장과 대시보드 요약 API를 제공하지 않아 플로우가 끝까지 이어지지 않았다.
2. **시도 2:** FastAPI에 `POST /api/actions`, `GET /api/analysis`, `GET /api/dashboard/summary`, `GET /api/validation/summary`를 추가해 UI가 필요한 읽기/쓰기 흐름을 보강했다.
   이 단계에서 브라우저 시연 흐름은 연결됐지만, 프론트 자동 테스트는 `ResizeObserver` 부재로 실패했다.
3. **시도 3:** Vitest + Testing Library를 추가하고, 테스트 셋업에 `ResizeObserver` mock을 넣어 JSDOM 환경에서 대시보드 차트가 렌더링되도록 조정했다.
   그 결과 빈 상태, 분석 중 버튼 비활성화, 조치 저장 성공 메시지까지 자동 테스트로 고정할 수 있었다.

## 최종 해결 (Resolution)
`#4`는 "화면 추가"가 아니라 "UI와 API를 묶은 데모 플로우 완성"으로 정리했다.

- `web/` 프론트엔드 앱 생성
- 입력, 분석 결과, 조치 기록, 대시보드 화면 구현
- FastAPI에 조치 저장/최근 이력/검증 요약 API 추가
- CORS 허용으로 로컬 프론트 개발 서버 연동
- Vitest 환경 구성과 `ResizeObserver` mock 추가
- `pytest 33개`, `vitest 3개`, `vite build` 통과 확인

## 배운 것 (Lessons Learned)
프론트엔드가 없는 저장소에서 UI MVP를 추가할 때는 화면 컴포넌트보다 먼저 "브라우저가 끝까지 호출할 수 있는 API 집합"을 맞춰야 한다. 또 차트 라이브러리를 도입하면 테스트 환경에서 필요한 브라우저 API가 무엇인지 같이 챙겨야 한다. 다음 단계의 E2E에서는 기능 자체보다 "실제 브라우저와 테스트 브라우저의 차이"를 먼저 점검하는 것이 낫다.
