# 개발 일지: E2E 검증 자동화와 발표용 요약 생성

**일자:** 2026-04-04
**관련 버전:** v0.1.0
**관련 역할:** build, qa

## 배경 (Context)
`#5` 이슈의 목표는 FactLog가 "코드가 있다" 수준이 아니라, 실제로 `입력 → 분석 → 조치 기록 → 대시보드 반영`이 끝까지 동작한다는 것을 자동화된 형태로 증명하는 것이었다. 채용 과제 특성상 데모 한 번 성공하는 것보다, 같은 플로우를 반복 검증할 수 있다는 점이 더 중요했다.

## 문제 (Problem)
`pytest`와 `vitest`만으로는 핵심 사용자 플로우가 실제 브라우저에서 끝까지 이어지는지 보장할 수 없었다. 추가로 E2E를 붙이자 테스트용 DB가 기존 로컬 DB와 섞일 위험이 있었고, Vitest가 Playwright 테스트와 `node_modules` 내부 테스트까지 주워 실행하려는 문제도 생겼다. 발표용으로 쓸 검증 수치도 코드 산출물과 별도로 정리할 필요가 있었다.

## 시도한 것들 (Attempts)
1. **시도 1:** `@playwright/test`를 추가하고 브라우저 E2E를 붙였다.
   기본 골격은 빠르게 잡혔지만, 테스트가 기존 SQLite 파일을 공유하면 실행 순서에 따라 결과가 흔들릴 수 있었다.
2. **시도 2:** `FACTLOG_DB_PATH` 환경변수를 도입해 E2E 전용 DB 파일을 `.tmp/factlog-e2e.db`로 분리했다.
   이로써 브라우저 테스트가 로컬 개발 DB를 오염시키지 않게 됐지만, Vitest가 `e2e/`와 외부 테스트 파일까지 주워 실행하는 문제가 남았다.
3. **시도 3:** `vitest.config.ts`의 include/exclude 범위를 `src/**/*.test.*`로 고정하고 `e2e/**`를 제외했다.
   프론트 단위 테스트와 브라우저 E2E를 분리할 수 있었고, Playwright는 핵심 플로우 1개만 안정적으로 검증하게 됐다.
4. **시도 4:** NASA 검증 산출물에서 발표용 요약을 재생성하는 스크립트를 추가했다.
   코드와 발표 자료 사이에 수작업 복붙 단계를 두지 않고, Markdown/JSON 산출물로 검증 결과를 재사용할 수 있게 됐다.

## 최종 해결 (Resolution)
`#5`는 테스트 추가가 아니라 "검증 결과를 반복 가능한 형태로 고정"하는 작업으로 정리했다.

- Playwright로 `입력 → 분석 → 조치 기록 → 대시보드 반영` 브라우저 플로우 자동화
- `FACTLOG_DB_PATH`로 E2E 전용 SQLite 경로 분리
- Vitest 범위를 축소해 Playwright 테스트와 충돌 제거
- `scripts/export_validation_summary.py` 추가로 발표용 검증 요약 JSON/Markdown 생성
- 최종 검증 상태 고정
  - `pytest 33 passed`
  - `vitest 3 passed`
  - `playwright 1 passed`
  - `vite build passed`

## 배운 것 (Lessons Learned)
브라우저 E2E를 붙일 때 가장 먼저 고정해야 하는 것은 테스트 시나리오보다 데이터 격리다. 테스트용 DB가 분리되지 않으면 성공 여부가 환경 상태에 종속된다. 또 발표용 수치는 문서에 손으로 적기보다 코드 산출물에서 다시 생성하도록 두는 편이 훨씬 안전하다. 다음에 AIHub 보조 검증을 붙일 때도 같은 방식으로 요약 산출물을 자동 생성하는 구조를 유지하는 것이 맞다.
