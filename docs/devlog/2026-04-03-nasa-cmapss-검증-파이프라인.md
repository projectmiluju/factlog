# 개발 일지: NASA C-MAPSS 검증 파이프라인 구축

**일자:** 2026-04-03
**관련 버전:** v0.1.0
**관련 역할:** build, qa, ship

## 배경 (Context)
FactLog의 첫 구현 이슈는 화면이 아니라 "이상 감지를 실제 데이터로 검증할 수 있는가"를 증명하는 것이었다. PRD에서도 NASA C-MAPSS를 메인 시연 데이터셋으로 사용하고, AI허브 데이터는 보조 검증으로 분리하기로 합의했다.

## 문제 (Problem)
초기 저장소에는 백엔드 코드, 데이터 처리 기준, 테스트 체계가 전혀 없었다. 이 상태에서 UI부터 만들면 채용 과제의 핵심인 "정량 검증"을 끝까지 연결하지 못할 위험이 컸다.

## 시도한 것들 (Attempts)
1. **시도 1:** NASA C-MAPSS를 그대로 RUL 예측 문제로 다루는 방향을 검토했다.
   발표 임팩트는 있었지만, MVP의 사용자 플로우는 "이상 여부 판단"에 가깝기 때문에 잔여수명 회귀는 화면/설명/대시보드와 연결 비용이 더 컸다.
2. **시도 2:** `RUL <= 30 cycle` 기준의 이진 분류로 단순화했다.
   이 방식은 이상 여부, 설명 문구, 추후 유사 사례 검색과 연결하기 쉬웠고, 정확도/정밀도/재현율/F1/ROC-AUC로 발표용 수치를 정리하기도 쉬웠다.
3. **시도 3:** 딥러닝 대신 `LogisticRegression(class_weight="balanced")`를 기준선으로 채택했다.
   2~3일 MVP에서 가장 중요한 것은 최고 성능이 아니라 재현 가능한 기준선과 빠른 실험 반복이라고 판단했다.

## 최종 해결 (Resolution)
`src/factlog_ml/` 패키지에 NASA C-MAPSS 로더, 라벨 생성 규칙, 기준선 모델, 검증 요약 생성 로직을 추가했다. `scripts/run_validation.py`로 재현 가능한 실행 경로를 만들고, `artifacts/validation/nasa_fd001_summary.json`에 요약을 저장하도록 고정했다. QA 단계에서는 파일 누락, 컬럼 수 불일치, 출력 디렉토리 자동 생성, AI허브 보조 검증 note 보존까지 자동 테스트로 보강했다.

## 배운 것 (Lessons Learned)
- 채용 과제 성격의 MVP에서는 모델 복잡도보다 검증 흐름의 완결성이 중요하다.
- 데이터 원본과 산출물 정책을 `.gitignore`에서 초기에 정리하지 않으면 기록과 커밋 이력이 쉽게 오염된다.
- AI허브 데이터는 제품 내부 완전 통합보다 "보조 검증"으로 위치를 낮추는 편이 현재 일정에서 더 현실적이다.
