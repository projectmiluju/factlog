# 개발 일지: 분석 API 구현과 조회 정합성 복구

**일자:** 2026-04-04
**관련 버전:** v0.1.0
**관련 역할:** build, qa

## 배경 (Context)
`#3` 이슈의 목표는 저장된 센서 레코드를 기준으로 이상 점수, 근거 설명, 유사 사례를 한 번에 반환하는 분석 API를 만드는 것이었다. 이번 단계가 완료되어야 FactLog의 핵심 흐름인 `입력 → 분석 → 근거 확인`이 백엔드 수준에서 이어진다.

## 문제 (Problem)
초기 구현은 `POST /api/analysis` 응답 자체는 만들었지만, 저장 후 다시 조회하는 단계에서 생성 시점 정보가 손실됐다. 특히 `explanation_source`는 상세 조회에서 `"stored"`로 하드코딩되었고, 유사 사례의 `similarity_score`는 조회 API에서 `0.0`으로 재작성되어 QA에서 반려됐다.

## 시도한 것들 (Attempts)
1. **시도 1:** `similar_case_ids`만 저장하고 조회 시 다시 조립하는 방식을 유지했다.
   생성 시점 payload를 최소화할 수 있다는 장점은 있었지만, 유사도 점수와 설명 출처처럼 파생 정보가 보존되지 않아 실패했다.
2. **시도 2:** `analysis_result` 테이블에 `explanation_source`와 `similar_cases_payload`를 추가하고, 조회 API는 저장된 payload를 그대로 복원하도록 변경했다.
   생성 응답과 조회 응답의 정합성을 맞출 수 있었고, 기존 로컬 DB를 깨지 않기 위해 간단한 `ALTER TABLE` 마이그레이션도 함께 넣었다.

## 최종 해결 (Resolution)
분석 결과 저장 구조를 "ID만 저장"에서 "조회에 필요한 최소 payload 저장"으로 바꿨다.

- `analysis_result`에 `explanation_source` 컬럼 추가
- `similar_cases_payload` 컬럼 추가
- 앱 시작 시 누락 컬럼을 보완하는 SQLite 마이그레이션 추가
- `GET /api/analysis/{id}`와 `GET /api/similar-cases/{id}`는 저장된 payload를 그대로 사용하도록 수정
- QA에서 추가한 회귀 테스트와 404 테스트까지 포함해 `pytest -q` 기준 30개 통과 확인

## 배운 것 (Lessons Learned)
분석 API처럼 "생성 시점 계산값"이 중요한 기능은 정규화만으로 끝내면 안 된다. 조회 단계에서 같은 의미를 복원해야 한다면, 계산 비용보다 정합성이 우선이며 필요한 파생값은 명시적으로 저장하는 편이 낫다. 다음 단계에서 조치 기록이나 대시보드 요약을 설계할 때도 "다시 계산해도 같은 값이 나오는가"를 먼저 따져야 한다.
