# 개발 일지: 입력 및 CSV 업로드 API 구현

**일자:** 2026-04-03
**관련 버전:** v0.1.0
**관련 역할:** build, qa

## 배경 (Context)
FactLog의 메인 사용자 플로우는 센서 데이터가 안정적으로 들어와야 시작된다. `#1`에서 NASA C-MAPSS 기준 검증 파이프라인을 만들었지만, 실제 제품 흐름인 "수기 입력 또는 CSV 업로드 → 저장 → 후속 분석"은 아직 비어 있었다. `#2`의 목적은 이 입력 계층을 FastAPI와 SQLite로 고정하는 것이었다.

## 문제 (Problem)
초기 저장소에는 API 프레임워크와 DB 계층이 없었다. 단순히 저장만 되게 만드는 것은 쉬웠지만, PRD가 요구한 핵심은 "잘못된 입력이 들어왔을 때 행/컬럼 단위로 설명 가능한 오류를 주는가"였다. QA 단계에서는 특히 `source_dataset` 허용값 검증과 빈 필수 문자열 필드 검증이 빠져 있다는 문제가 드러났다.

## 시도한 것들 (Attempts)
1. **시도 1:** 수기 입력과 CSV 업로드를 모두 같은 `SensorRecordCreate` 저장 모델로 수렴시켰다.
   저장 경로를 하나로 만들면 이후 `#3` 분석 API에서 입력 형태와 무관하게 같은 레코드를 참조할 수 있다고 판단했다.
2. **시도 2:** FastAPI의 `UploadFile`로 CSV 업로드를 구현했다.
   구현은 빨랐지만 런타임에서 `python-multipart` 의존성이 빠져 있어 즉시 실패했다. 의존성 선언을 보강하고 설치해 해결했다.
3. **시도 3:** QA가 추가한 엣지 케이스 테스트를 통해 입력 검증 구멍을 노출시켰다.
   `source_dataset="unsupported"`가 수기 입력 API에서 통과되고, 빈 `equipment_name`이 CSV 파서에서 저장되는 문제가 있었고, 각각 `Literal` 제한과 필수 문자열 공백 검증으로 수정했다.

## 최종 해결 (Resolution)
`src/factlog_ml/api.py`, `api_schemas.py`, `db.py`, `uploads.py`를 추가해 입력/업로드 계층을 구현했다. 수기 입력은 `POST /api/sensor-records`, CSV 업로드는 `POST /api/uploads/csv`로 분리했고, SQLite에 `equipment`, `sensor_record`, `analysis_result`, `action_log` 스키마를 만들었다. QA 이후에는 `source_dataset` 허용값을 스키마에서 강제하고, CSV 필수 문자열 필드가 비어 있으면 행/컬럼 단위 오류를 반환하도록 보강했다.

## 배운 것 (Lessons Learned)
- 입력 계층은 "저장 성공"보다 "잘못된 입력을 어떻게 설명하며 거절하는가"가 더 중요하다.
- FastAPI 파일 업로드는 `python-multipart`가 사실상 필수이므로 의존성 문서화가 누락되면 바로 깨진다.
- QA가 추가한 엣지 케이스를 바로 테스트 코드에 편입시키면, 구현보다 회귀 방지가 더 단단해진다.
